## Phase 1: テスト整備・品質保証（推定5-7日）

### 目的
実ユーザーテスト前の品質担保。致命的バグの早期発見とリグレッション防止。

### 1-1. E2Eテスト（Playwright）
**優先度: 最高**

ユーザーの主要動線をカバー:

| テスト | 内容 |
|--------|------|
| 認証フロー | ログイン→ニックネーム登録→ホーム遷移 |
| 店舗閲覧 | ホーム→探す→店舗詳細→ストーリー展開 |
| 推し店登録 | 店舗詳細→推し店追加→推しページ確認 |
| 共感タップ | ストーリー→感情タグ選択→カウント更新 |
| ダイアリー | 新規作成→写真→店舗選択→気持ち複数選択→保存→一覧確認 |
| ファンレター | ダイアリー→レター送信→店舗側受信確認 |
| 予約打診 | 店舗詳細→予約フォーム→送信→店舗側一覧表示 |
| 店舗ダッシュボード | ログイン→KPI確認→ストーリー管理→メッセージ配信 |
| AIインタビュー | 開始→会話→完了→ストーリー自動生成確認 |
| 管理者 | ログイン→申請審査→ユーザー権限変更 |

### 1-2. APIテスト（Vitest）
**優先度: 高**

全43エンドポイントの正常系・異常系:

| テスト対象 | 項目数 | 内容 |
|-----------|--------|------|
| 認証 | 3 | 未認証拒否、権限不足拒否、正常認証 |
| 消費者API | 19 | CRUD操作、バリデーション、RLS確認 |
| インタビューAPI | 4 | 開始→メッセージ→完了フロー |
| ダッシュボードAPI | 9 | 店舗権限チェック、データ取得 |
| 管理者API | 11 | admin権限チェック、CRUD操作 |

### 1-3. コンポーネントテスト（Vitest + Testing Library）

UIの主要コンポーネント:

- EmpathyTap（タップ→カウント更新）

### 1-4. アクセシビリティ・レスポンシブ確認

## Phase 2: 実ユーザーテスト準備（推定3-4日）

### 2-1. テストアカウント・デモデータ整備
- テスト用店舗アカウント 3-5件作成
- デモ用ストーリー・メニューデータ投入

## 全体スケジュール

```
Phase 1: テスト整備 .............. ✅ 完了（61テスト + E2E 3スイート）
Phase 2: ユーザーテスト準備 ...... 3-4日（← 次はここから）
Phase 3: ユーザーテスト実施 ...... 2週間
Phase 4: プロダクト改善 .......... 2-3週間（並行で継続）
Phase 5: ローンチ準備 ............ 1-2週間
                          合計: 約6-8週間
```

### 優先度マトリクス

| 緊急度↓ / 重要度→ | 高 | 中 | 低 |
|---|---|---|---|
| **高** | E2Eテスト、RLSテスト | APIテスト | コンポーネントテスト |
| **中** | デモデータ整備、エラー処理 | パフォーマンス最適化 | a11y改善 |
| **低** | セキュリティ確認 | バンドル分析 | リファクタリング |
